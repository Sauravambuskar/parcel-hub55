## Goal

Add an "Export to Excel" feature on the admin **Revenue Management** page that produces an accountant-ready `.xlsx` workbook with every order as a line item, fully split into Partner Payable, Platform Fee, GST and add-ons. Include an "All time (till date)" range so it can be used for historical bookkeeping.

## What the report will contain

**File:** `viasetu-accounts-report-<range>-<YYYYMMDD>.xlsx`

### Sheet 1 — Orders (line items)
One row per booking. Columns (in this order):

1. Order ID (`tracking_id` or short `id`)
2. Booking Date (`created_at`, IST, `DD/MM/YYYY`)
3. Booking Time (HH:mm)
4. Invoice Month (`MMM YYYY`) — for pivoting
5. Status
6. Payment Status (`paid` / `cop_pending` / `refunded`)
7. Razorpay Payment ID (`payment_id`)
8. Razorpay Order ID (`razorpay_order_id` if present)
9. Refund ID / Refund Amount (when applicable)
10. Courier Partner (`courier_name`)
11. Booking Source (`booking_source` — delhivery / xpressbees / shadowfax / shree_maruti / urbanebolt / prayog)
12. AWB (`prayog_awb` or partner AWB)
13. Sender Name, Sender City, Sender State, Sender Pincode, Sender GSTIN (if stored)
14. Receiver Name, Receiver City, Receiver State, Receiver Pincode
15. Goods Type
16. Dead Weight (kg), Chargeable Weight (kg), L×W×H (cm)
17. Shipment Value (declared)
18. **Partner Payable (₹)** = `base_fare − platform_fee` (fallback: `total − platform_fee − gst − packaging − insurance`)
19. **Platform Fee (₹)** = `platform_fee`
20. **Taxable Value (₹)** = Partner Payable + Platform Fee (i.e. `base_fare`)
21. **CGST (₹)** = GST/2 when sender state == "Karnataka" (ViaSetu's place of supply), else 0
22. **SGST (₹)** = same condition as CGST
23. **IGST (₹)** = full GST when sender state ≠ Karnataka, else 0
24. **GST Total (₹)** = `gst` (sanity column)
25. Packaging (₹), Insurance (₹)
26. **Grand Total (₹)** = `courier_price + packaging + insurance` (what customer paid)
27. Reconciliation Check — Excel formula `=ROUND(R-(P+Q+W+X+Y), 0)` should be 0; non-zero rows flag drift

Totals row at the bottom using `SUM()` formulas (not hardcoded), so accounts can filter and totals recompute.

### Sheet 2 — Monthly Summary
Pivot by Invoice Month: Orders, Gross Collections, Partner Payable, Platform Fee, CGST, SGST, IGST, Packaging, Insurance. All via `SUMIFS()` referencing Sheet 1, so it auto-updates if accounts edit.

### Sheet 3 — Partner Summary
Group by Courier Partner: Orders, Partner Payable (what we owe them), Platform Fee earned, Avg ticket size. Useful for partner-wise payout reconciliation.

### Sheet 4 — GST Summary
- Taxable Value, CGST, SGST, IGST, Total GST
- Split by Intra-state (Karnataka) vs Inter-state, suitable for GSTR-1 / GSTR-3B prep.

### Sheet 5 — Meta
Generated at, generated by (admin email), date range, row count, formula version, notes on definitions (Partner Payable, Platform Fee hidden inside Base Fare, GST split rule).

## UX changes on `src/pages/admin/RevenueManagement.tsx`

- Replace the current single "Export" button with a dropdown:
  - Export CSV (current)
  - **Export Excel (Accounts Report)** — new
- Add **"All time (till date)"** option to the date-range selector. This switches the export and the on-screen stats to lifetime data.

## Technical details

### Library
Use **`xlsx`** (SheetJS community, MIT) — already common, ~400 KB gzipped, supports formulas, multiple sheets, number formats. Install with `bun add xlsx`. Generate fully client-side; no server round-trip needed beyond data fetch.

### Data source
- For ranges Today/Week/Month/Year: reuse the existing `fetchBookings` result (already loaded, ≤2000 rows).
- For **All time**: add edge function `admin-export-bookings` (service-role) that:
  - Verifies caller is admin (`is_admin(auth.uid())` via JWT) **OR** accepts `x-prayog-auth` admin token consistent with other admin functions
  - Streams bookings in 1000-row pages ordered by `created_at desc`, returns the union as JSON
  - Selects only the columns needed for the export (no PII beyond what admin already sees in Revenue page)
  - This avoids the PostgREST 1000-row default cap and keeps a single source of truth for security.

### File generation
- New module `src/lib/accounts-export.ts` with `buildAccountsWorkbook(bookings, opts)` returning a `Blob`.
- Number cells written as numbers (not strings) with format `"#,##0;(#,##0);-"` and the Grand Total column highlighted.
- Header row: bold, frozen pane on row 1, autofilter enabled.
- Column widths set so Order ID, names, AWB are readable.

### GST split rule
Place-of-supply = Karnataka (ViaSetu HQ). Implemented as a single constant `PLACE_OF_SUPPLY_STATE = "Karnataka"` in the export module; if accounts later confirm a different rule we change one line. Document this in Sheet 5.

### Reconciliation invariant
Each row's `Reconciliation Check` formula makes drift visible to accounts. A conditional format (red fill when ≠ 0) is added so problematic rows pop out without manual inspection.

## Out of scope

- No DB schema changes. Pricing math is unchanged. Customer-facing invoices/UI unchanged.
- No automatic email of the report (can be added later if accounts wants a monthly cron).
- No PDF — Excel is what bookkeeping asked for.

## Validation

- Spot-check 3 known orders (e.g. `VIAS0000000017` ₹192 → Partner 75 + Platform 88 + GST 29 = 192) — sums must match Grand Total ±₹1.
- Open the file in Excel and Google Sheets; confirm formulas recalc, autofilter works, no `#REF!`.
- Lifetime export row count equals `SELECT count(*) FROM bookings`.
